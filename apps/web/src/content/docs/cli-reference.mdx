All commands support the global flags `--json`, `--quiet`, and `--yes`.

## epics init

Scaffold a new Epic package in the current directory. Creates `SKILL.md` and
`EPIC.md` with starter templates. New `SKILL.md` files use the `0.5.2`
dual-purpose shape, including the canonical `## Agent Epics` footer. Existing
files are preserved.

```bash
epics init
```

## epics upgrade-skill-footer

Refresh the canonical `## Agent Epics` footer in an authored `SKILL.md`.

```bash
epics upgrade-skill-footer
epics upgrade-skill-footer <path>
```

## epics install

Install an Epic from a repository path or local directory. The package is copied
into the host-local project path (e.g. `.claude/skills/<slug>/` for Claude).
CLI metadata is recorded in `.epics/installs.json`.

```bash
epics install <repo-path>
epics install --host claude <repo-path>
```

If `--host` is omitted in an interactive terminal, the CLI prompts for a
supported host. In non-interactive environments it fails instead of guessing.

## epics validate

Validate package metadata and supporting files. For `spec_version: 0.5.2`,
this includes the dual-purpose `SKILL.md` surface and canonical footer marker.
Returns diagnostics with level and path for each issue found.

```bash
epics validate
epics validate <path>
```

## epics info

Display package metadata and install record for an Epic. Accepts an optional
path or installed slug.

```bash
epics info
epics info <path-or-slug>
```

## epics status

Display a compact workflow summary for an Epic. Accepts an optional path or
installed slug and reports the current plan, next step, and latest log note.

```bash
epics status
epics status <path-or-slug>
```

## epics resume

Re-enter a long-running workflow without reconstructing context from scratch.
Builds a resume context from the package state, plans, and logs.

```bash
epics resume
epics resume <path-or-slug>
```

## epics doctor

Run workspace health checks. Reports status for managed metadata, authored Epic
presence, installed Epics, and local install source health.

```bash
epics doctor
```

## epics host setup

Configure the current workspace for a specific host. Creates host-specific files
and preserves any existing configuration.

```bash
epics host setup <host>
epics host setup claude
```

## epics daemon

Manage the shared per-user `epicsd` service and inspect runtime health.

```bash
epics daemon install
epics daemon uninstall
epics daemon start
epics daemon stop
epics daemon restart
epics daemon status
epics daemon logs
epics daemon doctor
```

`install` writes a user service for `launchd` on macOS or `systemd --user` on
Linux. `status`, `logs`, and `doctor` talk to the Unix-socket admin API.

## epics workspace

Register and inspect daemon-managed workspaces.

```bash
epics workspace register [path] --name repo-a
epics workspace list
epics workspace inspect <workspace-id>
```

## epics route

Create and manage daemon routes. V1 supports `webhook` and `cron` routes only.

```bash
epics route upsert --type webhook --workspace <workspace-id> --epic <slug> --provider github --endpoint repo-a --preferred-adapter claude --auth bearer --secret <token>
epics route upsert --type cron --workspace <workspace-id> --epic <slug> --job nightly --cron "0 2 * * *" --preferred-adapter claude --auth none --overlap queue_one
epics route list
epics route inspect <route-id>
epics route enable <route-id>
epics route disable <route-id>
```

Use `--id <route-id>` when replacing an existing route whose identity fields
change, for example when renaming a cron job or webhook endpoint key.

## epics run

Inspect the daemon run ledger.

```bash
epics run list
epics run list --route <route-id> --limit 20
epics run inspect <run-id>
```
