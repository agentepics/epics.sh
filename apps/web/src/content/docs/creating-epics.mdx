The [`create-epic`](https://github.com/agentepics/skills) skill is a guided
scaffolding tool for coding agents. Point your agent at it and it will walk
through a series of prompts to produce a spec-compliant Epic package ready for
use with the `epics` CLI.

## Invoking the skill

Skills are markdown instruction sets that agents follow. The `create-epic` skill
lives in the skills repo:

```
github.com/agentepics/skills/create-epic
```

Hand the skill source to your coding agent (for example via a slash command or
by pasting the file) and it will drive the scaffolding conversation.

## What gets created

The skill produces a standard Epic package layout:

```
my-epic/
├── SKILL.md        # Dual-purpose skill entrypoint plus Agent Epics footer
├── EPIC.md         # Human-readable overview and intent
├── runtime/
│   ├── state.json  # Persistent workflow state
│   ├── plans/      # Step-by-step execution plans
│   └── log/        # Session and event logs
├── cron.d/         # Scheduled tasks
├── hooks/          # Lifecycle hook handlers
└── policy.yml      # Constraints and guardrails
```

Every file has a role in the Epic lifecycle. The scaffolder fills in sensible
defaults so you can start iterating immediately.

This is the curated operational profile, not the minimum valid Epic. The core
spec only requires `SKILL.md` plus `EPIC.md`. If you want the smallest
spec-valid package, use `epics init`. If you want a fuller authored package
with runtime state, hooks, cron, and policy scaffolding, use `create-epic`.

## v0.5.2 migration note

New epics should emit a dual-purpose `SKILL.md` from day one: a short skill
surface at the top, and the canonical `## Agent Epics` footer at the bottom.
Migrating older epics usually means editing `SKILL.md`, not moving the runtime
tree again. For the full upgrade steps, see the public
[`0.5.2` migration guide](https://github.com/agentepics/agentepics/blob/main/docs/epic-migration-0-5-2.mdx).

## Archetypes

The skill asks you to pick an archetype that shapes the generated package:

- **Workflow** — A multi-step process with a clear start and end. Use this for
  implementation tracks, migrations, or release operations.
- **Instance** — A long-lived workspace that persists across sessions. Good for
  research, ongoing maintenance, or anything without a fixed finish line.
- **Capability** — A reusable building block that other Epics can compose. Pick
  this when the goal is a shared skill or utility rather than a standalone task.

## Key concepts

**Mutability tiers** — Not every file in an Epic has the same write rules. The
spec defines three tiers:

- **Working state** (agent-writable) — `runtime/state.json`, `runtime/log/`, plan progress.
  The agent updates these freely during execution.
- **Configuration** (human-only) — `policy.yml`, `cron.d/`, `hooks/`. Only a
  human or the scaffolder should modify these.
- **Immutable** (skill author) — `scripts/`, `references/`, and `assets/` stay
  fixed during normal operation.
- **Curated context** (rare updates) — `EPIC.md` carries the durable operating
  definition and should only change deliberately.
- **Skill/config surface** (setup-time updates) — `SKILL.md` is part of the
  authored package contract, but it is not an immutable runtime artifact.

**Cron schedules** — Epics can declare recurring tasks in `cron.d/`. These let
the agent perform periodic work like syncing state, running checks, or
publishing reports without manual triggers. The authored `0.5.2` profile uses
structured YAML like this:

```yaml
name: daily-sync
schedule: "0 9 * * *"
timezone: America/New_York
enabled: true
run:
  type: prompt
  prompt: |
    Resume the epic, review runtime/state.json, and update the active plan.
context:
  - runtime/state.json
output:
  log: true
  update_state: true
  notify: false
```

Fields: `name`, `schedule` (cron expression), `timezone`, `enabled`, structured
`run`, optional `context`, and structured `output`.

**Hooks** — Lifecycle hooks in `hooks/` fire at defined trigger points. The
canonical triggers are:

- `install` — after the Epic is installed into a workspace
- `uninstall` — before the Epic is removed
- `status-changed` — when `state.json` status field changes
- `milestone-complete` — when a plan milestone is marked done
- `plan-empty` — when all plan items are complete
- `became-blocked` — when the agent cannot proceed
- `cron-fired` — after a cron schedule executes

The current authored profile uses markdown prompt hooks with YAML frontmatter:

```markdown
---
enabled: true
type: prompt
timeout: 300
---

Read runtime/state.json, inspect the current plan, and write any required
follow-up to runtime/log/.
```

The broader spec also allows script, HTTP, and agent handlers. Current
`epics.sh` builds are strongest on install-time handling and daemon-backed cron
routes; package-native runtime support is still catching up to the full authored
surface.

**Policy constraints** — `policy.yml` sets guardrails for the agent. It has two
main sections:

- **`autonomy`** — spend limits, allowed tools, permitted network access, and
  file-system boundaries.
- **`escalation`** — rules for `on_error` (retry, pause, or notify) and
  `on_ambiguity` (ask human, pick default, or skip).

Policies keep autonomous work predictable without micromanagement.

**Validation** — The skills repo includes `validate-epic.sh`, a script that
checks package correctness: required files, schema compliance, hook
permissions, and cron syntax. Run it before publishing to catch issues early.
