`epicsd` now ships as a lean Phase B optional runtime sidecar. It is not
required for basic Epic usage — the CLI still handles install, validate,
resume, and doctor without it — but it now owns the first durable runtime slice
for shared background execution.

## Shipped V1 scope

- one shared daemon per OS user account
- Unix domain socket admin API at `~/.config/epicsd/epicsd.sock`
- localhost-only webhook ingress on `127.0.0.1`
- durable workspace, route, and run records under `~/.config/epicsd/`
- webhook and cron routes only
- one compiled-in runtime adapter at launch: `claude`
- no automatic cross-adapter failover
- no durable executor persistence
- no public remote control API

The daemon home layout is:

```text
config.json
epicsd.sock
state.json
workspaces.json
routes.json
runs/
secrets/
epicsd.log
```

## Runtime model

- route is the primary durable runtime object
- every webhook or cron event resolves directly to one route
- one bounded global ingress queue
- per-route single-flight execution
- per-workspace concurrency cap
- webhook dedup is in-memory best effort within the configured TTL
- cron overlap policies are `single_flight` and `queue_one`

## Security posture

- local admin control uses filesystem permissions on the Unix socket
- webhook ingress binds only to `127.0.0.1`
- webhook auth modes are `bearer` and `hmac`
- `authMode=none` is rejected unless explicitly enabled in daemon config
- routes fail closed when the workspace is degraded or the selected adapter is
  unhealthy

## Operator interface

The CLI surface now includes:

```bash
epics daemon install
epics daemon uninstall
epics daemon start
epics daemon stop
epics daemon restart
epics daemon status
epics daemon logs
epics daemon doctor
epics workspace register
epics workspace list
epics route upsert
epics route list
epics run list
```

`epics daemon install` writes a user service for `launchd` on macOS or
`systemd --user` on Linux.

## What it does not replace

The daemon does not replace host CLIs. It unifies the runtime layer underneath
them. An Epic package remains portable and valid without a daemon — `epicsd`
adds runtime capabilities on top of the portable baseline.
